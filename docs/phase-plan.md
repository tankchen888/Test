## 1. Phase 切分（建議）

### Phase 1 — MVP「能用、能信任」（目標：先讓既有屋主上線）
聚焦**現有屋主的剛性需求**：看得到自己的物業、收得到租金通知、找得到人、語言不再是障礙。

- **會員系統 + 實名認證（KYC）**：手機/帳號登入、護照/證件上傳、人工或第三方審核。
- **我的物業（綁定式）**：僅顯示綁定在本人名下之委託物件。
  - 房間現況（出租中/待出租、租約期間、月租金、管理方）
  - 租金到帳紀錄（結構化金額，不翻譯）
  - 合約與財報文件庫：**租賃合約 / 委託管理合約 / 代管合約 / 年度財報**（PDF，受權限保護、下載留軌跡）
- **統一客服收件匣 + 中↔泰即時翻譯**：整合 LINE / WhatsApp / 站內訊息為單一工作台；客服端與會員端各看自己語言、保留原文。
- **會員點數（累積與顯示）**：消費/租金回饋累點、點數明細、等級顯示。
  - *履約*：Phase 1 兌換**由專屬管家人工安排**（自營接送/SPA），尚不需線上商城。
- **在地資訊（CMS）**：泰國在地新聞與吃喝玩樂文章，後台可編輯。
- **雙入口**：微信小程序（中港）+ App（海外）共用後端 API；同帳號體系。

> Phase 1 不做：線上兌換商城、第三方商家平台、物件搜尋全功能、AI 管家。

### Phase 2 — 「有附加價值、會天天打開」
- **點數兌換商城**：自營權益上架 + 少量精選合作商家；線上兌換、核銷碼/QR、預約排程（接送、SPA、住宿、餐飲）。
- **在地服務預約系統**：行程、時段、管家指派、到店核銷。
- **物件搜尋與線上洽詢**：行銷物件庫、篩選、預約看屋/諮詢。
- **多城市內容擴充**：杜拜、倫敦等城市的在地內容與服務據點。
- **推薦裂變**：推薦好友獎勵、分享機制（小程序分享卡 / App 邀請碼）。

### Phase 3 — 「平台化與智能化」
- **合作商家平台規模化**：商家後台、結算對帳、評價。
- **資產儀表板**：跨物件報酬分析，串接既有 ERP / dashboard 數據。
- **會員體系深化**：跨城市權益互通、等級權益（如機場 VIP 通關）、生日/節慶禮遇自動化。
- **智能管家（AI）**：常見問答自動回覆、行程建議、文件問答。
- **開放 API**：與合作旅遊/金融服務串接。

## 2. 系統架構（高層）

```
        微信小程序 (中港)            App iOS/Android (海外)            後台管理 (Web)
              │                            │                              │
              └──────────────┬─────────────┘                             │
                             ▼                                            ▼
                    API Gateway / BFF  ── 身分驗證(KYC/JWT) · 權限 · 多語系 · 稽核軌跡
                             │
   ┌──────────┬─────────────┼──────────────┬───────────────┬────────────────┐
   ▼          ▼             ▼              ▼               ▼                ▼
 會員/KYC   物業/合約     點數/兌換     客服/翻譯       在地內容/CMS      服務預約/履約
   │          │             │              │               │                │
   └──────────┴─────────────┴──────────────┴───────────────┴────────────────┘
                             ▼
            資料層：PostgreSQL（交易） + 物件儲存(S3/OSS，文件) + 快取
                             ▼
        外部整合：翻譯 API · LINE/WhatsApp · 簡訊/推播 · 金流(僅必要) · ERP
```

**雙入口、單後端**：小程序與 App 走同一組 API，差異只在前端框架與分發管道，避免兩套邏輯。

## 4. 跨境合規 Checklist（務必早期釐清）

> 服務橫跨六國，個資、金流、稅務規範各不同，建議 Phase 1 即諮詢法務。

- **個資保護**：台灣《個資法》、歐盟 GDPR（英國）、泰國 PDPA、馬來西亞 PDPA、中國《個人資訊保護法》（小程序）。需隱私政策、同意機制、資料當地化評估。
- **跨境資料傳輸**：護照、合約等敏感資料的儲存地點與傳輸合法性。
- **點數法律性質**：點數**定位為회員回饋、非可儲值貨幣**，兌換不找零、不可提現，避免觸及電子支付/儲值牌照與洗錢防制。若未來涉及購買點數，需重新評估金流牌照。
- **KYC / AML**：高資產客戶實名認證與資金來源合規。
- **微信小程序合規**：需具備中國境內主體或服務商資質、類目審核（房地產/金融類目門檻高，須提前確認）。
- **稅務文件**：各國租金收入、財報的稅務揭露格式差異。
- **合作商家合約**：履約、退款、責任歸屬、發票/收據。

## 5. 履約模型（自營＋少量精選合作）

| 權益類型 | Phase 1 | Phase 2+ |
|----------|---------|----------|
| 機場接送 | **自營**（自家車隊/簽約司機），管家人工安排 | 線上預約 + 排班系統 |
| 私人管家/導遊/就醫陪同 | **自營**團隊 | 行程系統化 |
| SPA / 按摩 | 1–2 家**精選合作**，人工預約 | 多家上架 + 核銷 |
| 五星住宿 | 1–2 家**精選合作**（含自家 The Reserve 集團） | 房況串接 + 線上兌換 |
| 餐飲（米其林等） | 少量合作，人工訂位 | 商城上架 |

**好處**：品質可控、客訴可追、初期不需龐大商家系統；缺點是品項少、擴張慢——正好用 Phase 2 的商城與商家後台逐步補上。

## 6. 里程碑（粗估，待團隊資源校準）

| 階段 | 主要交付 | 粗估 |
|------|----------|------|
| **M0 對齊** | 原型確認、合規諮詢啟動、技術選型 | 2–3 週 |

> 時程取決於團隊規模與微信類目審核時間，建議 M0 先把**合規與小程序類目**這兩個最大的不確定性鎖定。

## 8. 立即可決策的事項

1. **微信小程序主體/類目**：是否已有中國境內主體或服務商？（最大時程風險）
2. **App 技術選型**：原生 vs 跨平台（建議 React Native / Flutter 以共用一套碼）。
3. **翻譯 API 供應商**：品質與成本（中↔泰、中↔英、中↔越…）。
4. **點數累點公式與等級門檻**：需與財會確認入帳與成本認列。
5. **既有屋主資料**：物業、合約、租金資料現況（散落 Excel/ERP？）決定遷移工作量。
